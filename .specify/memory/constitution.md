# ai 项目宪章

本仓库以 **AI 辅助编程** 与学习实践为主。本宪章约束规格、实现与安全基线；当与临时习惯冲突时，**以本文件为准**。

## Core Principles

### I. 规格优先与可追溯（Spec-First）

**规则**：有意义的功能或行为变更 MUST 先有可追溯的规格依据（例如 Spec Kit 流程中的 `spec.md` / `plan.md` / `tasks.md`，或等效文档）。实现与提交说明 MUST 能指向对应需求或任务条目。

**理由**：AI 生成代码速度快，缺少规格锚点易导致范围漂移与不可维护的“隐式需求”。

### II. 最小范围与单一职责（YAGNI）

**规则**：每个变更 MUST 只解决已批准范围内的问题；禁止为“顺便整理”而扩大改动面（无关重构、风格大改、未请求的依赖升级）。若确需额外复杂度，MUST 在实现计划中显式记录理由（可沿用计划模板中的 Complexity Tracking）。

**理由**：小步提交、窄范围变更更易审查、回滚与复现，也更适合人机协作。

### III. 安全、隐私与密钥（Non-Negotiable）

**规则**：密钥、令牌、会话 Cookie、生产连接串、私钥等 MUST NOT 进入版本库。MUST 使用环境变量或私密配置，并提供不含秘密的示例（如 `.env.example`）。对外部 API、用户数据或网络请求，MUST 按最小权限与最小暴露面设计；引入依赖前 MUST 评估许可证与维护状况。

**理由**：AI 辅助下更容易误粘贴敏感信息；本原则为硬性红线。

### IV. 可验证与可观测（Verify & Observe）

**规则**：交付 MUST 具备可检查的验收标准（自动化测试、脚本校验或明确的手工步骤）。引入或修改关键路径时，MUST 说明如何验证（测什么、命令是什么）。日志与错误信息 SHOULD 足以定位问题，避免静默失败。

**理由**：规格与代码之间需要可重复的验证闭环，避免“看起来能跑”。

### V. 清晰、一致与可审查（Clarity）

**规则**：命名、目录结构与错误处理 MUST 与现有代码库保持一致风格；对非显而易见的逻辑 MUST 用简短注释或文档说明意图。由 AI 起草的代码 MUST 经人类可读性审查（命名是否贴切、边界是否处理、是否与规格一致）。

**理由**：可读性是长期成本的主要杠杆；AI 输出不等于最终审查通过。

## 安全、隐私与依赖

- **数据**：默认不收集非必要个人数据；若处理敏感数据，MUST 在规格与计划中写明范围、留存与访问控制。
- **依赖**：新增或升级依赖 SHOULD 固定版本或锁文件；重大版本升级 MUST 在变更说明中注明动机与风险。
- **供应链**：优先使用可信来源；避免复制粘贴不可审计的二进制或混淆脚本作为构建步骤。

## 协作与工作流

- **分支与变更**：功能开发 SHOULD 使用独立分支；合并前 SHOULD 自检宪章各条（尤其 III、IV）。
- **文档**：用户可见行为或环境要求变化时，MUST 更新 `README.md` 或对应文档。
- **Spec Kit**：优先使用项目内已配置的 `/speckit-*` 技能推进 **宪章 → 规格 → 计划 → 任务 → 实现**；紧急修补也 SHOULD 事后补全规格或记录债务。

## Governance

- **效力**：本宪章优先于个人习惯与非书面约定；与宪章冲突的 PR SHOULD 先修订宪章或取得明确豁免并记录在案。
- **修订**：修订 MUST 更新本文末尾版本号与 `Last Amended` 日期；语义化版本——MAJOR：原则删除或重新定义；MINOR：新增原则或重大扩展；PATCH：措辞与非语义澄清。
- **合规**：审查时 SHOULD 对照「Core Principles」与「安全、隐私与依赖」做快速检查。
